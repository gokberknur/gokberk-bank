# Dogfooding findings

Friction and gaps found while building gökberk bank against `@gokberknur/design-system`. **Bank-repo only** —
this log is the feedback loop; it is not (yet) filed into the design-system backlog. Each item: what we hit,
where, and how we worked around it.

Status: `open` (worked around app-locally) · `resolved` (DS shipped a fix) · `wontfix` (intentional).

## Component gaps (no `gok-*` element exists)

These are filled by app-local composites under `src/lib/components/` (see `.planning/features/foundations/`
and ADR-004). Building each is a vote for a future DS primitive.

## Component-API friction

| # | Where | Finding | Workaround | Status |
|---|-------|---------|------------|--------|
| 8 | `gok-table` | `column.format(value, row)` returns a display **string**, so inline **sparkline cells** / custom cell renderers aren't first-class. Hit when designing portfolio/watchlist rows with mini-charts (V01, V05). | Render sparklines outside the grid, or a dedicated column slot if/when supported; revisit cell-renderer API | open |
| 9 | `gok-progress` | Determinate-only — no indeterminate mode. | Use `gok-spinner` for unknown-duration waits (per patterns.md) | wontfix (by design) |
| 11 | `gok-table` | **Rich cells.** Cells are strings only — a status can't render as a `gok-tag` (rule + icon + text) *in the grid*, so the "status by rule + icon, never colour alone" principle is only half-honoured there (the bare word "Pending"/"Settled"). A `renderCell` prop exists but returns a Lit `TemplateResult`/`Node`, unusable from Svelte without coupling to the DS render layer. Hit building the A02 ledger. | Word-only status in the grid; full shape-rule + word in the transaction drawer (where we own the markup). Consider a slot/snippet cell API or a first-class `badge`/`status` column type. | open |
| 12 | `gok-table` | **No row-activate event.** Opening a row uses `gok-selection-change` (select == open); there's no distinct "activate". After the detail drawer closes, the table still holds the row in its selection `Set`, so clicking the **same** row again emits no event and the drawer won't re-open. Hit on the A02 ledger → drawer. **Assessment-v1 follow-up:** the controlled-selection workaround is correct and verified — under a real click on the row's `gok-checkbox` host it opens *and* reopens (Lit drives `.checked` true→false as the parent clears `selectedId` on close), now guarded by `e2e/accounts/wallet-ledger.spec.ts`. Two things to know: (a) a naive "transient activation" (`preventDefault` on `gok-selection-change`, or clearing `selection` synchronously in the handler) **does not work** — `preventDefault` skips the table's `requestUpdate`, and even a `queueMicrotask` clear is defeated by Lit's dirty-check refusing to re-assert `.checked=false` after the user toggled the box natively; the controlled prop is the only reliable reset. (b) The activation is **checkbox-only** — there is no full-row click target and the select control is `tabindex="-1"`, so the drawer can't be opened by clicking the row body or by keyboard. (The assessment first read this as "drawer unreachable / no event fires", but that was synthetic/CDP clicks not triggering the custom checkbox — real clicks work.) | Drive selection as a **controlled** prop (`selection` via `setProps`) from app state and clear it on drawer close, so the same row reopens. A first-class **row-activate event** + a **full-row / keyboard-activatable** option would close the residual gap (otherwise the only fix is hand-building the grid, cf. #27). | open |
| 13 | `gok-segmented` | Not a form control — selection only via `value` + `change`, no native form participation. | Fine for the Direction/Density facets (read `value`, drive state); note if a `<form>`-bound segmented is ever needed. | wontfix (by design) |
| 14 | `gok-input` | **`type="number"` can't render grouped money** (`1,234.50`). Building the F07 money-input composite forced `type="text"` + `inputmode="decimal"` (keeps the numeric keypad, loses native number semantics). | Composite uses text + inputmode; canonical value is minor units. A first-class `gok-money` / numeric-format input would resolve it. | open |
| 15 | `gok-input` | **No caret-stable live grouping.** The inner `<input>` is in shadow DOM and unreachable, so an app-local composite can't restore the caret after grouping-while-typing. Grouping deferred to blur. | Live = ungrouped (natural caret); group on blur. Fix needs an exposed inner-input handle or a built-in formatter. | open |
| 16 | `gok-input` | **No font-feature / `--gok-input-*` hook** for tabular numerals on the value; relies on `font-variant-numeric` inheriting across the shadow boundary from the host. | Works as long as the inner input doesn't reset it; a `--gok-input-value-font` hook would be safer. | open |
| 17 | `gok-radio` / `gok-radio-group` | **No selectable card/tile variant, and selected state isn't stylable from light DOM.** Building the send-money recipient picker (P02) wanted payee "cards" with a selected-state border. The radio's `checked` doesn't reflect to a statically-prunable attribute, so `.payee[checked] { … }` is dropped by svelte-check as an unused selector; there's also no `--gok-radio-*` selected hook. | Wrap the host `gok-radio` with app-local card chrome (the radio control + rich multi-line label keep their own visuals) and drive the selected border from my own state via `class:is-selected={payee.id === draft.payeeId}` rather than the element's attribute. A first-class selectable-card variant or a reflected/`--gok-*`-exposed selected state would resolve it. | open |
| 18 | `gok-button` | **Only three variants — `primary`/`secondary`/`negative` — no quiet/tertiary/ghost.** Many flows want a *third*, quieter action alongside a primary + secondary: a reversibility "Cancel this payment", a "Clear all" filter reset, a disabled "Report a problem". An **invalid** variant (`ghost`) doesn't error — it fails `validateEnum` and **silently falls back to `primary`**, so a quiet action renders as a second loud accent button (caught visually on the send success screen — two filled-green buttons — not by `npm run check`). | Use `secondary` for quiet actions for now; a `tertiary`/`quiet` variant would let the hierarchy read in three levels. Lesson: an invalid enum prop on a `gok-*` element only warns at runtime, so lean on visual checks. **Recurred** on the A06 statements list (a subagent reached for `ghost` row actions → two solid-green buttons flooding a deliberately accent-free "document vault"), confirming this is a repeat footgun, not a one-off — a runtime dev-warning on an unknown variant, or an actual quiet/`ghost` tier, would stop it at the source. | open |
| 19 | `gok-input` | **No two-way value binding, and re-setting `.value` reactively risks a cursor reset.** Restoring a persisted wizard draft into fields (add-payee) needs an "uncontrolled initial value" idiom rather than a reactive `value` attribute. | An `initField` `{@attach}` that reads the draft through `untrack(...)` and sets each input's value **once** on mount. A documented uncontrolled-initial affordance would help. | open |
| 20 | `gok-icon` | **Icon set is thin for a fintech IA** (~23 glyphs) — no people/payees, request, split, exchange, scheduled, or direct-debit glyph. The payments-hub "Soon" tiles fall back to generic `plus`/`dash`/`settings`, so they read mostly from their text label. | Lean on labels; request a payments/finance glyph set (or allow a slotted custom SVG path) for the hub tiles. | open |
| 21 | `gok-segmented` | The `change` event detail is typed as a bare `Event` in the CEM (not `{ value }`), so the selected value is read off `e.target.value` with a cast. | Read `(e.target as HTMLElement & { value: string }).value`; a typed `CustomEvent<{ value }>` detail would be cleaner. | open |
| 23 | `gok-tag` | **No status/icon affordance.** Variants are `default`/`selected`/`readonly` only, and there's no leading-icon slot — so the "status by rule + icon + text, never colour alone" principle can't be expressed by the tag itself. Hit building the C01 card status tag (Active / Frozen / Expired). | Inline a small SVG glyph + the word into the tag's default slot (an `inline-flex` wrapper). A `status` variant set or a dedicated `icon` slot on `gok-tag` would make rule+icon+text first-class. Sibling of #11 (rich cells), for the tag. | open |
| 24 | `MoneyInput` (composite) | **No commit/blur callback** — `onchange` fires on every keystroke (input, paste, AND blur), so a per-change side effect (e.g. a "limit set" toast on the card-controls page) fires once per character. | Debounce the side effect (600ms). A dedicated `oncommit`/blur hook on the composite would be cleaner; flagging it on our own composite to fix when it's promoted. | open |
| 25 | `MoneyInput` (composite) | **Can't take an external/custom error** — validation is internal (balance/min/max only). The card daily-limit's reward-early "below what I've already spent today" rule had to live in a separate page-level message line, not the field's own reserved line. | Add an `error`/`validate` prop so a flow-supplied rule shares the one message line. (Our composite — fix on promotion.) | open |
| 26 | `MoneyInput` (composite) | **`value` is seed-only** — the display is seeded once via `untrack` at mount, so an external reset (e.g. "No daily limit" clearing the value) doesn't reflect without a `{#key}` remount. | `{#key}`-remount on external resets for now; a controlled-value sync would avoid the remount. (Our composite.) | open |
| 28 | `gok-card interactive` | **Whole-card link + an independent inner control is fragile.** The "stretched overlay `<a>`" idiom (one anchor `position:absolute; inset:0` over the card, the WalletCard pattern) makes a card one big link — but `gok-card` doesn't establish a containing block, so the overlay resolves against the nearest *positioned* ancestor and **escapes the card to blanket the page**; the last card's overlay then swallows clicks everywhere (hit on the M02 offers grid — the page "Redeem" button navigated to the last offer). And hosting a *separately*-clickable control (an Activate `gok-switch`) inside that card needs manual `z-index` layering + a click-stop. | Set `position: relative` on the host card (`.offer`) so the overlay is bounded to it; raise the inner control above the overlay (`z-index` + `stopPropagation`). A first-class "card as link with action slot" (or a documented requirement that an interactive card is a positioned containing block) would prevent the footgun. | resolved (app-side) |
| 27 | `gok-table` | **Can't host a dense grid that needs inline visuals per row.** The V01 portfolio holdings grid needs an embedded `Sparkline` (a live ECharts node) in every row plus rule+icon+sign delta cells — but `gok-table` cells are formatted **strings** only (`column.format` returns text; `renderCell` returns a Lit `Node`, unusable from Svelte; #8/#11). So the whole holdings grid had to be **hand-built** as an app-local accessible table (semantic `<table>`, `<th scope="col/row">`, `aria-sort` on sortable headers driven by app state, tabular numerals, hairline rules, row → `/invest/instrument/[symbol]` via an in-cell anchor for keyboard + a row `onclick` for mouse). A future `gok-table` would need a **per-cell slot/snippet renderer** (a framework-agnostic way to project an arbitrary node into a cell — e.g. a named-slot-per-column or a `column.cell` that yields a DOM node the host owns) plus a typed **column kind** for `sparkline`/`delta`/`badge` so rule+icon status and mini-charts are first-class without forking the DS render layer. Sibling of #8 (sparkline cells) and #11 (rich/status cells), proven out on the heaviest data screen in the app. | Hand-built app-local table for the holdings grid; keep `gok-table` for string-only ledgers (A02). Revisit if a slot/renderer cell API ships. | open |
| 29 | tokens | **Two `--gok-*` tokens referenced across the app don't exist in the published v0.4.1: `--gok-color-surface-sunken` and `--gok-radius-full`.** Unknown CSS custom properties fail **silently** — `background: var(--gok-color-surface-sunken)` (no fallback) resolves to an invalid value → transparent panel; `border-radius: var(--gok-radius-full)` → 0 → square corners. No build/`check` error catches it; it only showed up grepping the usages against the published token list (and visually). Was lurking in `payments/transfer`, `invest/OrderTicket`, `accounts/TransactionDrawer`, `money/RedeemFlow`. | Swapped to the real tokens — `--gok-color-surface-strong` (the recessed/inset surface) and `--gok-radius-pill` (full round). Confirms this is a recurring footgun — a lint of `var(--gok-*)` names against the shipped catalog would stop it at authoring time. | resolved (app-side) |
| 30 | (no `gok-slider`) | **No range/slider control.** Amount + term (loan apply) and term (mortgage calc) want a drag-to-set slider; the DS ships none. | App-local native `<input type="range">` styled with `--gok-*` (accent fill via a `--fill` gradient, token thumb, standard focus ring). Legitimate, but a `gok-slider` (single + range, token-themed, with a labelled read-out) is a real gap for lending/settings/filter surfaces. | open |
| 33 | `gok-dialog` (nesting) | **A nested `gok-dialog`/`StepUp` tears down its parent dialog.** `gok-dialog` emits **composed** `gok-close`/`gok-cancel` events that bubble through shadow boundaries. Invisible to `npm run check`; only shows when the flow dead-ends. | Guard the parent's close handler on `e.target === e.currentTarget` — event retargeting makes a bubbled event's `target` the *inner* dialog host, not the parent, so the parent only acts on its **own** dismissal. A first-class "only-my-own-dismissal" affordance (or non-composed close events) on `gok-dialog` would remove the footgun for anyone nesting dialogs. | resolved (app-side) |
| 34 | `gok-empty-state` | **No `success` (or status) variant.** Only `default`/`compact` ship, so a *successful-outcome* panel (the C02 "Card ordered / Replacement ordered" confirmation) can't signal success through the component — the variant is purely a layout switch, not a tone. Hit building the order-a-card success screen, which wants the empty-state's centred media+title+body+actions layout but with a clear positive read. | Use the `default` variant and carry the success signal by **composition** — an accent check glyph in the `media` slot + a status `gok-tag` ("Ready to use" / "On its way") + the copy — never colour alone. A `success`/status variant (or a `tone` prop) on `gok-empty-state` would make outcome panels first-class. Sibling of #23 (tag has no status affordance). | open |
| 35 | `toast` / `gok-toast` | **No action / undo affordance.** `toast(message, { status?, duration? })` carries only text — there's no slot or callback for an inline action, so the V05 "Removed AAPL · **Undo**" pattern (an undoable, low-stakes removal — the spec's intended affordance) can't be expressed. Hit on the watchlist row remove. | Plain "Removed {symbol}" toast; the removal stays reversible by re-adding via the combobox. | 36 | `gok-table` | **`rows` setter is reference-identity gated, so an in-place data mutation doesn't re-render.** The `rows` DOM property only re-renders when handed a **new array reference**; replacing a row immutably *inside* the same backing array (the repo's data-layer idiom — `arr[i] = {...arr[i], status}`) keeps the array reference, so the grid silently shows the stale cell. Hit cancelling a P07 request: the open-count header updated (a separate `$derived`) but the row's Status badge stayed "Open". Invisible to `npm run check` — only visible when a mutation doesn't paint. | Pass `rows` as a fresh array each time — `setProps({ rows: [...requests.all()] })` (or `$derived([...all()])`) so a mutation always yields a new reference. Sibling of the other `gok-table` items (#8/#11/#27 cells, #12 row-activate). | resolved (app-side) |
| 37 | `gok-dialog` over `gok-drawer` | **A `gok-dialog` opened while a `gok-drawer` is open is unclickable if authored as a sibling.** An open `gok-drawer` renders in the top layer (`popover="manual"` + `::backdrop`). Invisible to `npm run check` **and to a screenshot** (the modal renders correctly). | **Nest the confirm `gok-dialog` *inside* the `gok-drawer`** (as `RedeemFlow`/`OrderTicket` already do) so it shares the drawer's top layer and is clickable; pair with the #33 teardown guards (`target === currentTarget` + `preventDefault` while the confirm is open). A DS note that an open drawer's top layer occludes sibling dialogs — or a portal/top-layer opt-in on `gok-dialog` — would remove the footgun. | resolved (app-side) |

| 40 | `gok-button` | **`href` is silently ignored — `<gok-button href="…">` renders but never navigates.** The component has no `href`/anchor support (it's a `<button>`, not a link), but it accepts the attribute without warning, so an author reasonably expecting a link gets a dead control that looks fine and passes `npm run check`. ## Round 2 — surfaced while adopting `@gokberknur/design-system` 0.4.4

The 0.4.4 release resolved findings #1–#40. Adopting it back into the bank surfaced the next round of friction —
cases where the shipped primitive doesn't yet fit a real use-case. These are **new** feedback for a future DS round.

| # | component | finding | resolution | status |
|---|-----------|---------|------------|--------|
| 41 | `gok-slider` | **The shipped slider (0.4.4, answering #30/P5) can't express a rich, formatted slider.** Its `show-value` read-out prints the **raw numeric value**, so a money slider over integer minor units shows `1000000`, not `€10,000` — there is no value **format/renderer** hook. It also has no **min/max bound-rail labels**, and `label` always renders **visibly** with no external-label escape (`aria-labelledby` / accessible-label-only), so a slider whose design is *label + formatted value on one row, track, then a min–max rail* cannot adopt it without either regressing the read-out to a raw number or duplicating the label. Hit migrating the loan amount/term and mortgage-term sliders (Phase 2 of the 0.4.4 adoption). | Kept the app-local tokened `<input type="range">` (formatted `<output>`, min/max rail, `--fill` gradient, `aria-valuetext`) for these three controls — `gok-slider` adoption deferred. A `format` callback (or value-renderer slot), optional min/max bound labels, and a label-less/`aria-labelledby` mode would let rich money/term sliders adopt it. | open |
| 42 | `gok-money` | **No "available balance" affordance.** `gok-money` (0.4.4, answering #4/P1) has generic `min`/`max` bounds, but a bank's most common amount-field rule is "you can't send more than your **available balance**", which wants its own reward-early, no-blame copy — *"Not enough balance — you have €6,893.93"* — not the generic `max` copy *"Enter €6,893.93 or less"*. There is no `balance`/`available` prop, so the balance message and its formatted available-amount must be computed by the consumer and passed through `error`. Also surfaced: `gok-money` does **not** put the currency in the accessible name (it uses `label` verbatim), so the consumer must append it (`label="Amount (EUR)"`) to keep the field's name currency-aware. Hit across all 21 amount fields when adopting `gok-money`. | Kept `MoneyInput` as a thin (~40-line) adapter over `gok-money` that computes the balance error → `error` prop and appends the currency to the label. A first-class `balance` (or `available`) prop with built-in reward-early "you have X" copy, plus an option to fold the currency into the a11y name, would let the adapter disappear entirely. | open |
| 43 | `gok-combobox` / `gok-multi-select` | **Label-only options + select-only commit block the two real combobox surfaces.** Both (0.4.4, answering #F10) are data-driven on `options: { value, label, disabled? }[]` — **label is a single string**, with no sublabel/meta/icon fields, no per-option **slot or render hook**, and the row renders as *check + label* only. Two app surfaces can't adopt without regressing: **(a)** `invest/AddInstruments` is a catalog picker whose rows are *symbol (mono) · name · type badge · right-aligned price* — flattening each instrument to one `label` string drops the price/type columns a wealth picker is chosen by; and it's a **browse-and-batch-commit dialog** (tall listbox + "Add (N)" footer), not an inline chip field. **(b)** `payments/split`'s people picker filters on name **and** handle (foldable into the label) but has a free-text **"add by name"** escape (`split.addByName` mints an ad-hoc participant) — `gok-combobox` is explicitly **select-only** (free text reverts, never commits), so that path is lost; and it adds people **one-at-a-time into an external share table** with per-row amount/percent inputs, not as in-field chips, so `gok-multi-select`'s chips-in-field model doesn't match either. | Kept both hand-built WAI-ARIA composites (`AddInstruments`, the inline split picker) app-local; `gok-combobox`/`gok-multi-select` adoption deferred. What would unblock them: a rich-option shape (optional `description`/meta/`icon`) **or** an option-render slot; a `freeSolo`/`allow-create` mode that commits typed text; and (for split) an "add to external collection" mode rather than only in-field chips. | open |
